Rafal Wilinski · Founding Engineer, Runlayer

Speaker notes

Hi everyone, I'm Rafal. Today I want to talk about self-improving agents, but not in an offline single-player setting where just one agent gets better by itself. Imagine

Speaker notes

Imagine the best engineer you've ever worked with — the one who, every time they cracked something genuinely hard, wrote down exactly how they did it and handed it to the whole team. 04 / term
Working definition

Skill = a playbook an agent can load when the work starts.

Available skills · summaries only
oncall-runbook
chromium-lambda
pr-review
LLM context window
chromium-lambda loaded into context
Speaker notes

To make sure that we're on the same page. In this talk, a skill is essentially a playbook that an agent or a client can load when it thinks that knowledge is relevant. The technical pattern here is called progressive disclosure. In a simplified way, a skill consists of two things: a description or summary, and the actual content. At the beginning, the harness or client sees only the summary. But when it thinks the knowledge inside can be valuable, it can read the whole skill, insert it into context, and change its trajectory based on that acquired knowledge.

05 / METR
Long-horizon agents

Agents are getting better at staying on task.

Source: METR Task-Completion Time Horizons of Frontier AI Models, updated May 8, 2026
Speaker notes

Why are skills important? If you have not been living under a rock for the past six to twelve months, your workflow has changed dramatically, and you can see that in a quantifiable way. The METR benchmark measures task horizon: how long agents can work independently while still having a good chance of completing some task — with emphasis on the word chance. The trend is clear: frontier agents are getting better at longer, deeper, multi-step work. Caveat: only if those frontier capabilities are not taken from you. God I miss Fable so much.

06 / deeper wrong paths
The hidden cost

A longer-running agent can dig a much deeper hole.

More autonomy means more useful work, but also more time to over-invest in a local optimum.

Speaker notes

But there is a catch. With great power, or long task horizons, comes great responsibility. If an agent can work independently for longer, it can also go much deeper into the rabbit hole. It can find a local optimum, spend hundreds of tool calls exploring one direction, defending it, and still not arrive at the right conclusion. The next agent can do the same thing. It can repeat the wrong exploration, or even find a worse one. So the initial trajectory matters a lot. That is why skills are important: they can guide the agent toward the right target. Otherwise you waste time, tokens and most importantly, money.

07 / enterprise examples
Why enterprises care

The valuable skills are the rituals nobody wants rediscovered.

“Have we seen this OOM before?”

Root cause, fix, PR, and what not to restart blindly.

“How does this vendor get approved?”

SAP screens, exception path, budget code, approval order.

“Which deploy path is real?”

Legacy script, runtime flags, environment variables, rollback.

“Can I say this to the customer?”

Severity proof, safe language, escalation owner, approval gate.

Speaker notes

The most valuable knowledge is often not glamorous. It is tribal ritual that is probably embedded deeply inside your company, and if your agents try to discover that from first principles, like SAP approval paths, legacy deployment scripts, customer escalation language, and things like that, without a curated playbook, they will likely fail. And many times, that failure can be really costly. Effectively, that is why I believe so many AI POCs are failing across the industry. You're not giving the agents the relevant context, the playbooks to act accordingly within your company.

08 / ecosystem problem
The current ecosystem

Problem #1:Skills are local-first and dev-centric.

That is fine for developers. It breaks when a whole company needs shared, updated knowledge.

Open Hamel Husain tweet about non-developers sharing skills with a team and keeping everyone updated
Reference: Hamel Husain on X
Speaker notes

Skills are great, but they are not problem-free. I think there are three important problems standing in the way of making them great. The first one is that they are local-first and dev-centric. When you look at skills today, the things you mostly find are CLI commands, Git repositories, config files as JSON, and so on. That works great if you are a developer and if this is single-player mode. But if you are a company, or an AI adoption leader, or a VP of AI trying to roll out an AI strategy across the whole company, that approach does not scale. So the challenge is: how can non-technical people use and share skills?

Speaker notes

Btw, this is what happens when your junior developer discovers npx skills. That's the only command I know that installs a playbook, three prompt injections, and someone else's API keys and untrusted root-access scripts, all in one go. We're scanning thousands of skills every day, and trust me — there's wild stuff in there.

12 / skills over MCP
Skills over MCP

Make the remote MCP server the source of truth for skills.

Source: github.com/modelcontextprotocol/experimental-ext-skills
Speaker notes

Instead of installing skills locally, the remote MCP server can become the source of truth. It can search the right skills, enforce policy, and make sure everyone gets the freshest, best version of the skill. We have been doing this at Runlayer for over six months already, but this is no longer just our internal thing — we see the broader MCP ecosystem going in this direction too.

13 / ext-skills

14 / API
Technically simple

Two functions get you surprisingly far.

Search for relevant skills. Fetch the actual skill file. The harder work is ranking, scoping, governance, and evaluation.

findSkills({
  query: "debug chromium lambda fork"
}) → SkillSummary[]

getSkillFile({
  skillName: "chromium-lambda-runtime",
  fileName: "SKILL.md"
}) → string
Speaker notes

While the official spec is leaning towards MCP Resources, we decided to go with a very simple API surface composed of just these two tools — you can get surprisingly far. One function finds relevant skills for the query. The other is for fetching skill files. The API is minimal. The hard parts are everything around it: ranking, scoping, versioning, auditing, and all the governance work that makes this useful in a company.

16 / self-improving agents

Self-Improving Agents That Teach the Company Back

Speaker notes

Ok, so we now know how we can improve agents — using skills — and we can leverage MCP to distribute them to agents. But let's talk about something more interesting: how can agents create and improve those skills for themselves?

17 / Voyager
This was visible in 2023

Voyager turned exploration into a growing skill library.

In Minecraft, the agent explored, wrote reusable code skills, retrieved them later, and improved without changing model weights.

Source: Voyager, arXiv:2305.16291
Speaker notes

Whenever I see a new paper or framework about self-improving agents, I always go back to the Voyager paper. This was back in 2023, ages ago, right? Right after the GPT-4 release. The question was simple: how long can an agent explore independently, and what can it learn? Voyager introduced this brilliant idea of a skill library. When the agent figured out how to do something useful, like craft a pickaxe, build a furnace, or make a crafting table, it saved the recipe for how it achieved that.

18 / skill library
The mechanism

Exploration becomes reusable procedural knowledge.

Source: Voyager, arXiv:2305.16291
Speaker notes

How? The key thing to notice is the skill library. Voyager did not just explore Minecraft and throw the trace away. When the agent achieved something, it saved the recipe as a list of tool calls for how it got there. Later, when the same capability was needed again, the agent could retrieve that persisted skill instead of reasoning from scratch. That is the core idea I want to carry forward: exploration should turn into reusable procedural knowledge.

19 / distill
The learning loop

Take the successful run. Distill the reusable part. Draft a skill.

Speaker notes

And we started doing something very similar and straightforward. Let the agent do real work. When the run succeeds, take the trace and ask another LLM to distill the reusable part. Then draft or update a skill.

20 / iteration
Self-improvement

Every run — pass or fail — distills a sharper skill.

Successes and failures both feed the distiller — which ships the next revision of the skill straight back to the agent.
Speaker notes

Distilling a skill from a clean success is the easy half — we learn the most from mistakes; the real sharpening happens when the agent fails. A run that goes sideways is the richest signal you have: it exposes the missing guardrail, the edge case, the thing nobody wrote down. So every run, pass or fail, feeds the distiller. The wins confirm what already works and make the skill more trustworthy; the failures are what actually make the skill more robust. All that happens completely autonomously and async, with no human in the loop.

24 / teach the company back

Self-Improving Agents That Teach the Company Back

Speaker notes

How can we do it? Well — we have all the pieces of the puzzle. We have Skills as a means to improve agents, retain knowledge, and guide models. We have MCP as the unified distribution layer for those skills — and also the central repository, the knowledge base, for all of them. And we have the continuous-learning flywheel that makes one agent smarter. The interesting part — and this is the network effect I promised at the start — is what happens when you multiply that by every agent, on every client, in the entire company.

25 / flywheel
Putting it together

Self-improving organization flywheel.

Speaker notes

At Runlayer we call this the self-improving organization flywheel. Any agent, on any client, does some work. A bunch of similar runs gets distilled into a skill. It passes through the governance layer — scanning for prompt injections and PII, with a human in the loop — and then MCP serves it back to every client. In the middle, we have a constantly evolving knowledge base — always up to date, continuously updated with new alpha, fed on new edge cases. So one agent solves something once — and every other agent, in every department, starts the next task already knowing it. Nobody re-derives that crazy incantation or OOM playbook ever again. Your company gets smarter in the background, while you sleep. That is, very literally, teaching the company back.

25 / boundary
Distribution failure

A bad skill shared across agents becomes organizational hallucination.

Speaker notes

But there's also a catch — and it's a big one. A hallucinating agent is a bad day. A hallucinating skill is a policy. One bad agent makes one bad call; one bad skill makes that same bad call on every client, every day, automatically. You didn't ship a bug — you promoted it to company policy and gave it a distribution channel. That's the real reason a bad skill is worse than no skill: with no skill the agent thinks for itself; with a bad one it's confidently wrong, fast, and now it can teach the new hires. So scope, evidence, and rollback aren't enterprise checkboxes — they're the line between organizational learning and organizational hallucination.

MCP

One distribution layer.

A remote MCP server unifies, centralizes, governs, and makes skill usage transparent.

Self-learning

Agents distill their own skills.

From real successes and failures — not from someone writing docs by hand.

Flywheel

Promote once, everyone benefits.

MCP serves the skill across the whole company — it gets better while you sleep.

Speaker notes

These are the four things I'd want you to walk out with. Skills make your agents better — and let you keep what they've learned, even when the model underneath changes. A remote MCP server is the distribution layer: one place to unify, govern, and actually see how skills get used. Agents can distill those skills themselves, from real successes and failures, instead of waiting for someone to hand-write the docs. And the flywheel spreads one agent's win across the entire company. Put them together and here's what changes: when an agent finishes, the company is smarter than it was before — not just another task done, but knowledge that compounds, every run.

27 / moat
Everyone's asking what their moat is

The model is rented. The knowledge it leaves behind is your moat.

Speaker notes

Right now, everyone in our industry is asking the same question: the models themselves have basically become a commodity, cost of software is going to zero, everyone can rebuild my product, at least from the surface — so what is actually my moat? That is the one thing a competitor can't copy by swapping in a better model or by burning more tokens than you.
