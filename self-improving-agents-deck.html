04 / term
Working definition

Context

When to use it

The trigger conditions that make this playbook relevant.

Taste

What good looks like

Speaker notes

Before we go further, I want to make sure we are all on the same page. In this talk, a skill is essentially a playbook that an agent or a client can load when it thinks that knowledge is relevant. The technical pattern here is called progressive disclosure. In a simplified way, a skill consists of two things: a description or summary, and the actual content. At the beginning, the harness or client sees only the summary. But when it thinks the knowledge inside can be valuable, it can read the whole skill, insert it into context, and change its trajectory based on that acquired knowledge.

05 / METR
Long-horizon agents

Agents are getting better at staying on task.

METR tracks the human task duration at which an agent is predicted to succeed with a target reliability.

Source: METR Task-Completion Time Horizons of Frontier AI Models, updated May 8, 2026
Speaker notes

Why are skills important? If you have not been living under a rock for the past six to twelve months, your workflow has probably changed pretty dramatically, and you can see that in a quantifiable way. This benchmark measures task horizon: how long agents can work independently while still having a good chance of completing some task. I think the word "chance" is essential here, because agents are not deterministic, they are probabilistic. And you can see that the horizon keeps improving. Every time we feel like we hit a wall, there is a new model, a new harness, or a new agent, and suddenly it works better. The trend is clear: frontier agents are getting better at longer, deeper, multi-step work. Caveat: only if those frontier capabilities are not taken from you. More on that later.

06 / deeper wrong paths
The hidden cost

A longer-running agent can dig a much deeper hole.

More autonomy means more useful work, but also more time to over-invest in a local optimum.

Speaker notes

But there is a catch. With great power comes great responsibility. If an agent can work independently for longer, it can also go much deeper into the wrong path. It can find a local optimum, spend hundreds of tool calls exploring one direction, defending it, and still not arrive at the right conclusion. The next agent can do the same thing. It can repeat the wrong exploration, or even find a worse one. So the initial trajectory matters a lot. That is why skills are important: they can guide the agent toward the right target. Otherwise you waste time, you waste money, you get bad tool calls, bad emails, and bad side effects.

08 / ecosystem problem
The current ecosystem

Problem #1:Skills are local-first and dev-centric.

That is fine for developers. It breaks when a whole company needs shared, updated knowledge.

Open Hamel Husain tweet about non-developers sharing skills with a team and keeping everyone updated
Reference: Hamel Husain on X
Speaker notes

Skills are great, but they are not problem-free. I think there are three important problems standing in the way of making them great. The first one is that they are local-first and dev-centric. When you look at skills today, the things you mostly find are CLI commands, Git repositories as source of truth, local files, plugins, client config, and JSON files. That works great if you are a developer and if this is single-player mode. But if you are a company, or an AI adoption leader, or a VP of AI trying to roll out an AI strategy across the whole company, that approach does not scale. So the question is: how can people share skills with their team, and how can you make sure everyone automatically gets the best updated version of that skill?

14 / API
Technically simple

Two functions get you surprisingly far.

Search for relevant skills. Fetch the actual skill file. The harder work is ranking, scoping, governance, and evaluation.

findSkills({
  query: "debug chromium lambda fork"
}) → SkillSummary[]

getSkillFile({
  skillName: "chromium-lambda-runtime",
  fileName: "SKILL.md"
}) → string
Speaker notes

We decided to go with a very simple API surface. With just these two functions, you can get surprisingly far. One function finds relevant skills for the query. Another function fetches the relevant skill file. The API is minimal. The hard parts are everything around it: ranking, scoping, versioning, auditing, and all the governance work that makes this useful in a company.

18 / Voyager
This was visible in 2023

Voyager turned exploration into a growing skill library.

In Minecraft, the agent explored, wrote reusable code skills, retrieved them later, and improved without changing model weights.

Source: Voyager, arXiv:2305.16291
Speaker notes

And of course, we can all author skills by hand, but that part is not the interesting part. Everyone can do that. What I think is really interesting is the Voyager paper. This was back in 2023, right after GPT-4 was released, when people were still trying to measure agent effectiveness in harnesses like Minecraft: how long can an agent explore independently, and what can it learn? Voyager introduced this brilliant idea of a skill library. When the agent figured out how to do something useful, like craft a pickaxe, build a furnace, or make a crafting table, it saved the recipe for how it achieved that. Then, in the future, it did not have to rediscover the same thing from first principles. It could retrieve the skill, almost like a higher-order function or reusable code, and use that persisted knowledge directly.

19 / skill library
The mechanism

Exploration becomes reusable procedural knowledge.

Source: Voyager, arXiv:2305.16291
Speaker notes

So while this diagram is on screen, the key thing to notice is the skill library. Voyager did not just explore Minecraft and throw the trace away. When the agent achieved something, it saved the recipe for how it got there. Later, when the same capability was needed again, the agent could retrieve that persisted skill instead of reasoning from scratch. That is the core idea I want to carry forward: exploration should turn into reusable procedural knowledge.

20 / distill
The learning loop

Speaker notes

The loop we started using is straightforward. Let the agent do real work. When the run succeeds, take the trace and ask another LLM to distill the reusable part. Then draft or update a skill. Some agent frameworks, like Hermes-style self-learning agents, do similar things. The important part for us is that the output is not hidden memory; it is a reviewable, editable artifact.

25 / boundary
Distribution failure

A bad skill shared across agents becomes organizational hallucination.

Speaker notes

Here is the sharper version of the failure mode. A bad skill shared across agents becomes organizational hallucination. It is no longer one agent making one bad call. It is the company repeatedly teaching agents the wrong procedure. That is why scope, evaluation, and rollback are not enterprise checkboxes. They are the difference between organizational learning and organizational hallucination.

27 / iteration
Iteration

Skills improve like code: patches, evidence, rollback.

Each successful run is a signal. Some diffs move the skill toward better task completion. Some need to be rejected or rolled back.

Source: SkillGrad, arXiv:2605.27760
Speaker notes

That is why versioning matters. Zero-shot skills are often not good enough. As agents uncover more edge cases, the system should patch the existing skill. You can think of this as gradient descent over skills: each run gives you a signal about the procedure. But some patches will make the skill worse, so skills need to behave more like code: versions, diffs, rollback, and eventually A/B tests between the current skill and a candidate update.
