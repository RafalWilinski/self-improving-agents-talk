AI Engineer World's Fair 2026

Self-Improving Agents That Teach the Company Back

Rafal Wilinski · Founding Engineer, Runlayer

Speaker notes

Hi everyone, I'm Rafal. Today I want to talk about self-improving agents, but not as an offline single-player loop where one agent just gets better by itself. I want to talk about how we can apply that self-improving loop across a whole organization, and ideally inside an enterprise. When an agent does useful work, it should leave behind a better playbook for the next run, the next person, and eventually the whole company.

Speaker notes

Before we go deeper, I want to make sure we are all on the same page about the word "skill." In this talk, a skill is essentially a playbook that an agent or client can load when it thinks that playbook is relevant. The technical pattern here is progressive disclosure. A skill has two parts: the actual content, and a short summary or description. At the beginning, the agent only sees the summary. When it thinks, "huh, this seems relevant," it decides to read the full skill and inject that content into its context. That is how the agent acquires the knowledge it needs for the task.

03 / METR
Long-horizon agents

Agents are getting better at staying on task.

METR tracks the human task duration at which an agent is predicted to succeed with a target reliability.

Source: METR Task-Completion Time Horizons of Frontier AI Models, updated May 8, 2026
Speaker notes

Okay, so why are skills important now? If you have not been living under a stone for the last six to twelve months, your workflow has probably changed pretty dramatically. And we can actually see that in a quantified way. There is this benchmark from METR that measures task horizon: roughly, how long can an agent work independently on a task and still have a good chance of completing it? The word "chance" is essential here, because these systems are probabilistic, not deterministic. As you can see, that horizon keeps improving. Every time it feels like we hit a wall, we wait a month or two, a new model shows up, and suddenly agents can work for longer again. The trend is clear: frontier agents are getting better at longer, deeper, multi-step work. Caveat: if those frontier capabilities are not taken from you. More on that later.

04 / deeper wrong paths
The hidden cost

A longer-running agent can dig a much deeper hole.

More autonomy means more useful work, but also more time to over-invest in a local optimum.

Speaker notes

But there is a catch. With great power comes great responsibility. If an agent can work independently for longer, it can also go much deeper into the wrong path. It can find a local optimum, spend hundreds of tool calls defending it, and then when the next agent starts from scratch, it repeats the same exploration, or maybe even finds a worse path. So the initial trajectory matters a lot. Being able to guide the agent toward the right target is extremely important, because otherwise you waste time, you waste money, and sometimes you optimize for the wrong thing entirely. This is what makes persistent guidance, using skills for example, so valuable.

18 / API
Technically simple

Two functions get you surprisingly far.

Search for relevant skills. Fetch the actual skill file. The harder work is ranking, scoping, governance, and evaluation.

findSkills({
  query: "debug chromium lambda fork"
}) → SkillSummary[]

getSkillFile({
  skillName: "chromium-lambda-runtime",
  fileName: "SKILL.md"
}) → string
Speaker notes

From a technical point of view, the basic protocol can be tiny. One function finds relevant skills for a query. Another function fetches the skill file. The hard part is not the JSON schema. The hard part is ranking, scoping, versioning, auditing, and deciding which skills are allowed to influence which agents.

05 / Voyager
This was visible in 2023

Voyager turned exploration into a growing skill library.

In Minecraft, the agent explored, wrote reusable code skills, retrieved them later, and improved without changing model weights.

Source: Voyager, arXiv:2305.16291
Speaker notes

And of course, we can all author skills by hand, but that part is not the interesting part. Everyone can do that. What I think is really interesting is the Voyager paper. This was back in 2023, right after GPT-4 was released, when people were still trying to measure agent effectiveness in harnesses like Minecraft: how long can an agent explore independently, and what can it learn? Voyager introduced this brilliant idea of a skill library. When the agent figured out how to do something useful, like craft a pickaxe, build a furnace, or make a crafting table, it saved the recipe for how it achieved that. Then, in the future, it did not have to rediscover the same thing from first principles. It could retrieve the skill, almost like a higher-order function or reusable code, and use that persisted knowledge directly.

09 / distill
The learning loop

Take the successful run. Distill the reusable part. Draft a skill.

Speaker notes

The loop we started using is straightforward. Let the agent do real work. When the run succeeds, take the trace and ask another LLM to distill the reusable part. Then draft or update a skill. Some agent frameworks, like Hermes-style self-learning agents, do similar things. The important part for us is that the output is not hidden memory; it is a reviewable, editable artifact.

21 / boundary
Distribution failure

A bad skill shared across agents becomes organizational hallucination.

Speaker notes

Here is the sharper version of the failure mode. A bad skill shared across agents becomes organizational hallucination. It is no longer one agent making one bad call. It is the company repeatedly teaching agents the wrong procedure. That is why scope, evaluation, and rollback are not enterprise checkboxes. They are the difference between organizational learning and organizational hallucination.

23 / iteration
Iteration

Skills improve like code: patches, evidence, rollback.

Each successful run is a signal. Some diffs move the skill toward better task completion. Some need to be rejected or rolled back.

Source: SkillGrad, arXiv:2605.27760
Speaker notes

That is why versioning matters. Zero-shot skills are often not good enough. As agents uncover more edge cases, the system should patch the existing skill. You can think of this as gradient descent over skills: each run gives you a signal about the procedure. But some patches will make the skill worse, so skills need to behave more like code: versions, diffs, rollback, and eventually A/B tests between the current skill and a candidate update.
